# Writable Inventory and Equipment

## Summary

Build inventory support in staged layers. Catalog inspection and
save-generation comparison remain read-only. The normal TUI now exposes a
full staged inventory editor; new records use conservative catalog-derived
defaults and every write is source-bound, backed up, atomically replaced, and
reparsed.

## Domain and Format Work

- Complete the inventory model by identifying its outer counts, boundaries,
  category ordering, equipment references, and category-specific payloads.
- Verify bags, consumables, gear, weapons, companions, furniture, scrap, and
  sleeping bags using IL2CPP metadata, the item database, and existing saves.
- Document each verified layout and retain captured payload bytes unchanged
  for existing items.
- Add domain types such as `ItemDefinition`, `InventoryItem`,
  `EquipmentState`, `InventorySnapshot`, and staged `InventoryChange`
  operations. `ItemDefinition`, `InventoryItem`, and `InventorySnapshot` are
  complete for the read-only phase.
- Add `load_item_catalog(game_dir)` using pinned UnityPy `1.25.x`; the installed
  `ResourcesBundle` uses LZ4HC and cannot be handled by an LZMA-only parser.
- Add `parse_inventory()`, `read_inventory()`, and write APIs independent of
  Textual. Parsing and reading are complete. The general final-state writer
  rebuilds the equipment, saved-bags, and carried-inventory region from
  staged raw records with exact-source and reparse verification.
- Keep catalog properties distinct from unverified save-format properties.
  Do not infer stack limits, record defaults, or reference identity from the
  item database.

## Mutation Rules

- Stage all changes and apply them in one atomic write with one timestamped
  backup.
- Merge additions into existing carried records when item ID and serialized
  type match.
- Construct new instances from catalog-derived type/category data with
  zero-filled unknown payload bytes and leave them unequipped.
- Delete records when removing items and update all enclosing counts and
  lengths.
- Allow positive signed 32-bit carried quantities and reject zero or negative
  retained quantities.
- Support equipment slots for hat, jacket, trousers, shoes, bags, weapon, and
  companion, using catalog compatibility rules.
- Equipping a replacement returns the displaced item to carried inventory.
- Removing an equipped item automatically stages its unequip operation and
  clearly reports that consequence.
- Quick slots, death items, buybacks, stashes, and world containers remain
  outside the edited region.
- Reconstruct only the verified inventory/equipment region, preserving
  unknown payloads and all unrelated prefix/suffix bytes.
- Before writing, re-read the save and reject stale staged changes if its hash
  changed.
- Reparse the generated bytes and verify item counts, quantities, equipment
  references, ordering, and unchanged unrelated data before atomic
  replacement.
- Use the existing backup, verification, rollback, and external
  backup-directory guarantees.
- Reject configured backup directories inside the game installation or save
  tree before any inventory write is enabled.

## TUI

- Add an Inventory action and `i` binding alongside the existing
  character-value editor.
- Open a responsive inventory screen with carried items, equipment slots,
  name/ID/category search, quantities, and staged-change indicators.
- Provide keyboard and mouse actions for Add, Remove, Change Quantity, Equip,
  Unequip, Apply, and Discard.
- Use a searchable catalog modal for additions and request quantity for every
  added item.
- Show a final confirmation summarizing additions, deletions, quantity
  changes, and equipment changes.
- Prompt before discarding pending changes when leaving the inventory screen.
- After success, refresh the selected save, preserve its UUID selection, clear
  staged state, and display the backup path.
- Show specific errors when parsing, catalog loading, defaults, references, or
  final reparse verification fail.

## Tests

- Add synthetic fixtures for every item category and equipment slot; do not
  include real saves.
- Test exact round-trip serialization, malformed counts, truncation, unknown
  types, duplicate records, category payloads, and unsupported references.
- Test quantity boundaries, stack merging, new default instances, deletion
  and count updates, sorting, equip replacement, automatic unequip, and
  stale-file rejection.
- Assert backups contain the original, writes are atomic, rollback works, and
  bytes outside the reconstructed region remain identical.
- Add synthetic catalog fixtures covering item definitions, localization,
  corrupt JSON, missing assets, and UnityPy minor-version mismatches.
- Add Textual headless tests for searching, filtering, add/remove/edit/equip
  flows, staged cancellation, confirmation, failures, backup reporting, and
  wide/narrow layouts.
- Run the complete unittest suite, syntax checks, editable install, console
  entry point, and wheel build.

## Assumptions

- "Inventory" covers carried inventory and equipment only; quick slots, death
  items, buybacks, and external stashes remain unchanged.
- Newly constructed records use conservative defaults rather than verified
  game-generated defaults for every category.
- The installed game bundle is the authoritative item catalog, with numeric
  fallback labels for display only.
- Inventory edits require the game to be closed, and the persistent
  close-game warning remains mandatory.
- Controlled in-game validation may modify only slot1. Stop at explicit
  checkpoints for the user to close or launch the game.
